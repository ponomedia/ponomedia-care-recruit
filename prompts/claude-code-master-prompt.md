## 制作方針

### 基本姿勢
- モックに忠実に再現する。「雰囲気を参考に」ではなく「完全再現」が目標
- スマートフォンファースト（375px 基準）
- セマンティック HTML を使用し、SEO・アクセシビリティを確保する
- 外部フォント・ライブラリは CDN 経由で読み込む（Google Fonts: Noto Sans JP）
- 画像は `/images/` ディレクトリに格納。モックに画像がある場合は適切な alt テキストを付与

### 技術スタック
- HTML5 / CSS3（Flexbox・Grid）
- Vanilla JS（軽量実装を優先）
- Schema.org 構造化データ（JobPosting・LocalBusiness）
- OGP メタタグ完備
- Google Tag Manager タグ埋め込みスロット（body 直後）

## LP 実装ルール

### モック再現に関するルール
1. モック画像に記載されているコピー・テキストはそのまま使用する（意訳・要約禁止）
2. カラーはスポイトで抽出し、HEX で指定する
3. レイアウト・余白・フォントサイズの比率をモックから読み取って再現する
4. セクションの順序を変更しない
5. モックに存在しない要素を勝手に追加しない

### 禁止表現（法務・景表法対応）
- 「採用保証」「必ず採用できる」「採用率○○%保証」などの断定的成果表現
- 「業界最安値」「No.1」などの根拠のない最上級表現
- 個人の成果を一般的成果として表現すること

### コード品質ルール
- インデントはスペース 2 つ
- CSS はコンポーネント単位でコメントを入れる
- JS は `DOMContentLoaded` 内に記述し、グローバル汚染を防ぐ
- 画像の `loading="lazy"` を必ず付与
- フォームには `autocomplete` 属性を付与

## Codex 連携方法

### レビュー依頼手順
1. HTML ファイルが完成したら `codex-review-prompt.md` を Codex に渡す
2. Codex からのレビュー結果を受け取り、指摘事項を修正する
3. 再度レビューを依頼し、総合評価 90 点以上になるまで繰り返す

### ファイル受け渡しルール
- コードはコードブロックで共有する
- モック画像は「モック_[ファイル名]」の命名で添付する
- レビュー結果は `reviews/` フォルダに `YYYY-MM-DD_review.md` で保存する

## 禁止事項

- モックを無視した独自デザインの追加
- 法務リスクのある表現の使用
- 依頼されていない機能の勝手な実装
- `.env` ファイル・API キーのハードコーディング
- 外部サービスへの不要なデータ送信コードの埋め込み
- 「参考として」モックを扱うこと（完全再現が原則）
